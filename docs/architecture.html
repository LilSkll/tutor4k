🔄 Жизненный цикл запроса к репетитору (Tutor Request Lifecycle)
1
Клиент: Пользователь пишет «В чём разница ser и estar?» → fetch POST /api/tutor
2
Middleware: AUTH Проверка Supabase-сессии → загрузка профиля (level, language, userName)
3
Cache Check: CACHE Нормализация запроса → SHA-хэш → поиск в tutor_cache → попадание? → мгновенный ответ, 0 запросов к LLM
4
Domain Guard: isOffTopic()? → «Как приготовить борщ?» → отказ без вызова LLM
5
RAG Retrieval: retrieveContext("ser estar", level=A1) → FTS по 4710 чанкам → топ-5 фрагментов из Дышлевой/Гонсалес-Алимовой
6
System Prompt: buildSystemPrompt({ level: A1, retrievedContext, userName }) → промпт на русском + материал учебников + правила сократического метода
7
Provider Chain: LLM Groq ключ #1 → 200 OK → ответ (или 429 → ключ #2 → ключ #3 → Gemini) RETRY
8
Persist: Ответ сохраняется в tutor_cache (для будущих запросов) + chat_messages + recordStudySession (+2 минуты к streak)
9
Клиент: Markdown-рендеринг ответа → таблицы спряжений, примеры, «📖 Лексика урока» (A1/A2)
🔒
Изоляция и безопасность пользователей
Row-Level Security · Supabase Auth · Service-Role для записи
🔐 Row-Level Security
Каждая таблица защищена: USING (auth.uid() = user_id). Пользователь видит и пишет ТОЛЬКО свои данные. Даже при ошибке в коде БД откажет в доступе к чужим строкам.
👥 Supabase Auth
JWT-сессия в cookies. middleware.ts проверяет каждый запрос. UUID-идентификация. Триггер handle_new_user авто-создаёт профиль при регистрации.
⚙️ Service-Role Writes
Прогресс и история записываются через service-role клиент (обход RLS), но ID пользователя берётся из getUser() — идентификация проверяется, даже если cookies нестабильны.
